🎓 Spec Driven Development

Une méthodologie où la spécification est le point de vérité unique du projet. Le code en découle et doit être traçable jusqu'à la spec.

Pourquoi SDD ?

Le problème : un cahier des charges classique est ambigu, incomplet, interprétable. Un agent IA (ou un développeur) ne peut pas implémenter correctement sans poser des dizaines de questions.

SDD produit un document exécutable :

  • Suffisamment précis pour qu'un agent IA implémente sans ambiguïté
  • Suffisamment clair pour qu'un décideur non technique comprenne ce qui sera livré
"Un agent IA qui lit uniquement ce document peut-il implémenter correctement ce comportement, sans deviner ?" — Si non, la spec est incomplète.

SDD vs approches classiques

ApprocheProblèmeSDD
Cahier des chargesAmbigu, interprétableCas d'utilisation précis avec critères vérifiables
User StoriesTrop haut niveau, pas de flux détailléÉtapes acteur/système, exceptions, règles de gestion
Pas de specLe code EST la spec (indéchiffrable)La spec est la source, le code en découle

🖊️ Étape 1 — Spécification

Entrée : Notes informelles, cahier des charges, interviews

Sortie : docs/SPEC-racine-*.md

Skill : sdd-uc-spec-write (automatique)

La spécification est structurée par cas d'utilisation (UC). Chaque UC décrit une interaction complète entre un acteur et le système.

Concepts clés

ConceptRôleIdentifiant
Cas d'utilisationScénario acteur → systèmeUC-001
Étapes Na/NbAlternance acteur (a) / système (b)1a, 1b, 2a...
ExceptionCe qui peut mal tournerRattachée à une étape
Règle de gestionContrainte métierRG-0001
Critère d'acceptationSoit/Quand/Alors — vérifiableCA-UC-001-01
PackageRegroupement de UCNiveau 2 ≈ Epic, Niveau 1 ≈ Feature
Exemple : UC-001 — Créer un compte utilisateur

Acteur : Visiteur

État initial : Le visiteur n'a pas de compte

État final : Le compte est créé, un email de confirmation est envoyé

Étapes (cas nominal)

#DirectionDescription
1a→ ActeurLe visiteur saisit email, mot de passe, nom
1b← SystèmeLe système valide les données (RG-0001, RG-0002)
2b← SystèmeLe système crée le compte et envoie un email de confirmation

Exceptions

ÉtapeConditionRéaction
1bEmail déjà utiliséMessage "Un compte existe déjà avec cet email"

Critères d'acceptation

  • CA-UC-001-01 : Soit un email non enregistré, Quand le visiteur soumet le formulaire, Alors le compte est créé et un email est envoyé.
  • CA-UC-001-02 : Soit un email déjà enregistré, Quand le visiteur soumet le formulaire, Alors un message d'erreur s'affiche.

📐 Étape 2 — Conception technique

Entrée : docs/SPEC-racine-*.md

Sortie : 4 documents dans docs/

Skill : sdd-uc-system-design (automatique)

DocumentQuestion
ARCHITECTURE.mdComment le système est-il construit ?
DEPLOYMENT.mdComment le système est-il déployé et opéré ?
SECURITY.mdComment le système est-il sécurisé ?
COMPLIANCE_MATRIX.mdComment le système est-il conforme ? (si applicable)

Principe fondamental : traçabilité

Chaque décision d'architecture est rattachée à un UC-xxx, une RG-xxxx, une ENF-xxx ou un CA-UC-xxx-yy de la spec. Pas de décision sans justification.

Types d'architecture supportés

  • SaaS — multi-tenant, scalabilité, CDN
  • Client lourd / Desktop — distribution, mise à jour, mode hors-ligne
  • Driver / Firmware — certification, installation silencieuse
  • Embarqué — mobile, TPE (avec HAL), terminal professionnel, dev board

🗺️ Étape 3 — Planification

Entrée : docs/SPEC-racine-*.md + docs/ARCHITECTURE.md

Sortie : plan/lot-NN-nom.md

Skill : /sdd-plan (invocation manuelle)

Le développement est découpé en lots. Chaque lot :

  • Regroupe des UC liés (même package ou relations include/extend)
  • Est ordonné par dépendances (fondations d'abord)
  • Est testable de bout en bout une fois implémenté
  • A ses critères d'acceptation issus de la spec

Principes de découpage

#PrincipeExemple
1Dépendances d'abordAuth avant gestion des droits
2Cohérence fonctionnelleUC liés dans le même lot
3Taille raisonnable2 à 5 UC par lot, 3 à 8 lots
4PrioritéUC Critiques en premier
5TestabilitéChaque lot compile et se teste seul

🏗️ Étape 4 — Développement

Entrée : plan/lot-NN-nom.md + docs de conception

Sortie : src/, tests/, plan mis à jour

Skill : /sdd-dev-workflow <lot> (invocation manuelle)

La boucle

✏️
Tests d'abord
Traduire les CA en tests unitaires
⚙️
Implémenter
Coder selon ARCHITECTURE.md et SECURITY.md
✅
Vérifier les AC
Chaque critère est satisfait ou non
🔄
Itérer si nécessaire
Boucle numérotée jusqu'à 100% des AC

Reprise après QA 🔧

Si la QA échoue, le dev-workflow détecte automatiquement le rapport QA et bascule en mode reprise : il ne corrige que les scénarios en échec et les points de revue de code, sans refaire tout le lot.

🧪 Étape 5 — QA

Entrée : Lot implémenté (AC à 100%)

Sortie : qa/ (plan de test, revue de code, rapport)

Skill : /sdd-qa-workflow <lot> (invocation manuelle)

Processus

📋
Plan de test
Scénarios nominaux, limites, erreurs — soumis au pilote
▶️
Exécution
Tests automatisés + manuels si nécessaire
🔍
Revue de code
Lisibilité, sécurité, performance, conventions
📊
Rapport
Verdict : ✅ VALIDÉ ou ❌ À CORRIGER

Si le verdict est ❌ À CORRIGER, retour au développement via /sdd-dev-workflow en mode reprise (🔧).

Synthèse

Le pipeline complet

SDD = Spécifier → Concevoir → Planifier → Construire → Vérifier → Livrer

Chaque étape produit un livrable traçable.

Chaque livrable est exploitable par un agent IA.

Le code est la conséquence de la spec, pas l'inverse.
